# Agent guidance

This repo is moving through a v2 rewrite. The old file/YAML workbench is useful reference material, but it is not the target architecture.

Optimize for a reliable source-backed data workbench that can fetch, normalize, review, resolve, and release DC government structure/source data. Do not optimize for preserving the current record layout.

## Working style

- Use TDD for changes to persistence, source parsing, resolution replay, review behavior, and release output.
- Keep commits small and descriptive.
- Prefer behavior-oriented tests over snapshot tests of incidental formatting.
- Run the relevant tests before committing. Run the full suite before major handoff.
- When working on interactive CLI flows, test them from a person's perspective too. Use scripted stdin where possible and do at least one manual smoke pass for meaningful flows.
- Leave short PR/issue notes for important design deviations or source discoveries.

## Architecture rules

- SQLite is the v2 workbench/index/review layer.
- Source artifacts live as files and are indexed into SQLite.
- Resolution JSONL files record replayable human/workbench decisions.
- Releases are generated CSV/JSON/SQLite packages with one short README.
- Canonical entities and relationships should be compact and skimmable.
- Raw source rows belong in source item/evidence storage, not embedded in primary entity exports.
- Store one directed relationship fact. Generate incoming/backlink views for display/query.

## Avoid

- Do not create new YAML record types.
- Do not treat YAML files as the canonical database.
- Do not add broad new source coverage while implementing core v2 architecture unless the issue explicitly asks for it.
- Do not write long generic Markdown docs.
- Do not add separate release files for gaps, caveats, update pipelines, or deliverable tracking.
- Do not silently guess when source data is ambiguous. Preserve evidence and create review items.
- Do not make CI depend on live external source scraping by default.

## Source handling

At source boundaries, scraped/fetched data is untrusted. Validate or normalize it before inserting into typed workbench tables.

Keep source capture practical. A source may produce a page artifact, schema artifact, sample rows, full rows, document index, text archive, entity candidates, relationship candidates, dataset inventory rows, legal refs, or some combination. Do not force every source into the same depth.

## Resolutions

Resolution logs should be generated by the CLI whenever possible. A resolution should describe a decision, not just a text diff.

Useful resolution operations include accepting/rejecting candidates, merging candidates, setting canonical fields, accepting/rejecting relationships, fixing relationship type/direction, and resolving/defering review items.

Resolution replay should be deterministic. Conflicts should fail loudly and return the user to review.

## Review UX

The main review path should be interactive and TUI-ish:

```bash
deno task dc -- review
```

The user should be able to see evidence, accept the likely action, edit, reject, defer, or quit without composing long commands. Scriptable commands are useful secondary paths, not the main experience.

## Release UX

The release should be short and boring:

```text
README.md
manifest.json
dcgov.sqlite
entities.csv/json
relationships.csv/json
sources.csv/json
datasets.csv/json
legal_refs.csv/json
```

If the release needs a review-status note, put it in README.md. Avoid language like "gaps" or "caveats" in the release surface unless there is no clearer wording.

## Type safety

Prefer explicit domain types and typed database access. Keep loose `unknown`/`Record<string, unknown>` at source boundaries, then validate into typed rows.

Good boundaries:

```text
src/domain/       IDs, domain types, validators, pure operations
src/db/           migrations, schema bootstrap, typed query helpers
src/sources/      connectors and parsers
src/resolutions/  operation types and replay
src/review/       review item generation and interactive flow
src/release/      release builders/exporters
src/cli/          command wiring only
```

If a simpler structure is clearly better while implementing, use it and explain the decision briefly.
